# スキルの使いどころ

個人スキル（`~/.cursor/skills/`）のうち、**設計〜実装〜検証**に関わるスキルの使い分けガイド。

---

## スキル一覧（フェーズ別）

### 設計・探索（実装前）

| スキル | 一言 | いつ |
|--------|------|------|
| `/brainstorming` | 文脈把握 → 2〜3案 → 設計書 → `writing-plans` へ | アイデアがまだ粗い・新機能の起点 |
| `/brainstorming-persona` | PM・セキュリティ・UX など**複数視点**で盲点を出す | 視点をガラッと変えたいとき |
| `/brainstorming-devils` | 提案を**徹底批判**してリスクと反証を出す | 案は好きだが壊れないか殴りたいとき |
| `/grill-me` | だいたい決まったプランを**1問ずつ**詰めて合意する | 抜け・矛盾を潰すインタビュー |

### 計画・実装・検証

| スキル | 一言 | いつ |
|--------|------|------|
| `/writing-plans` | 実装プランを `docs/plans/` に書く | 設計が固まった後 |
| `/test-driven-development` | RED → GREEN → REFACTOR | コードを書くとき |
| `/webapp-testing` | ブラウザ MCP / Playwright で UI 検証 | 画面・E2E の確認 |

---

## 結論：定番パイプライン

設計スキルは**同じではない**。目的で選び、**複合**する。

**新機能（中〜大）の定番:**

```
/brainstorming
  → （任意）/brainstorming-persona
  → （任意）/brainstorming-devils
  → （任意）/grill-me
  → /writing-plans
  → /test-driven-development
  → （UI あれば）/webapp-testing
```

**急ぎ・仕様確定済み:**

```
/writing-plans  →  /test-driven-development
```

**小バグ:**

```
/test-driven-development だけ
```

---

## 設計4種の違い

| | brainstorming | persona | devils | grill-me |
|---|:---:|:---:|:---:|:---:|
| 方向 | 発散→収束 | 多視点発散 | 批判・反証 | 協調的深掘り |
| 起点 | ふわっとしたアイデア | ふわっとしたアイデア | **守りたい提案** | **だいたい決まったプラン** |
| 出力 | 設計書 + writing-plans へ | 盲点サマリ | 残リスクレポート | 共有理解サマリ |
| トーン | ファシリテーター | 役割プレイ | 敵対的 | インタビュアー |

- **brainstorming** だけが設計承認後に **writing-plans へ自動ハンドオフ**する本流
- **persona / devils** は探索・ストレステスト用。終わったら brainstorming・grill-me・writing-plans のどれかへ
- **grill-me** は「推奨案つきで一緒に決める」。devils は「わざと反対する」

## おすすめの複合フロー

### パターン1: 新機能（しっかり）

```
/brainstorming
「在庫アラート機能を追加したい」

→ 盲点が不安なら

/brainstorming-persona
「さっきの案を5視点で見て」

→ 方向は決まったが不安なら

/brainstorming-devils
「この設計を殴って」

→ 細部の穴埋め

/grill-me
「決定木の抜けを詰めて」

→ 実装準備

/writing-plans
```

### パターン2: アイデアはあるが視点が足りない

```
/brainstorming-persona  →  /brainstorming
```

多視点で広げてから、本流 brainstorming で2〜3案比較と設計書化。

### パターン3: 設計はほぼ固まっている

```
/brainstorming-devils  →  /grill-me  →  /writing-plans
```

最終ストレステスト → 穴埋め → プラン化。

### パターン4: 仕様書・Issue が明確

```
/writing-plans  →  /test-driven-development
```

brainstorming 系はスキップ可。

### パターン5: 小さなバグ修正

```
/test-driven-development
```

### パターン6: UI だけ確認

```
/webapp-testing
```

ロジック起因なら先に TDD。

## 実際の打ち方（例）

### 段階的（おすすめ）

```
/brainstorming
「ユーザー登録フローを追加。メール必須、OAuth は後回し」

→ 設計承認後（brainstorming が writing-plans を起動）

→ 実装

「Task 1 を /test-driven-development で」

→ UI

「登録画面を /webapp-testing で確認」
```

### 1発指定

```
/brainstorming
仕様: （アイデアを貼る）
制約: 承認後は writing-plans、実装は TDD、UI は webapp-testing
```

---

## やらなくていいとき

| 状況 | 理由 |
|------|------|
| 1行修正・typo | 設計スキル全部は重い |
| 設定ファイルだけ | TDD も例外確認が必要 |
| 捨てプロトタイプ | brainstorming / grill-me は時間がかかる |
| 提案がまだないのに devils | 殴る対象がない → brainstorming へ |

---

## 注意（トレードオフ）

| ポイント | 内容 |
|----------|------|
| brainstorming は長い | 承認ゲート + 設計書。急ぎは writing-plans 直行 |
| persona は発散向き | 収束は brainstorming か grill-me に任せる |
| devils は敵対的 | 協調的に詰めたいなら grill-me |
| grill-me は遅い | 1問ずつ。急ぎは writing-plans |
| writing-plans はファイルを残す | `docs/plans/` が増える |
| TDD は厳しい | `/test-driven-development` を明示すると守られやすい |
| webapp-testing は重い | ロジックは TDD、UI だけ E2E |
